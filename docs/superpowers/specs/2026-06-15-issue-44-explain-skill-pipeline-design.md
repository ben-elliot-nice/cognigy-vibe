# Design: Explain Skill Structure + MCP Build Pipeline (Issue #44)

## Goal

Prove the end-to-end architecture for externalising `explain` topic content from hardcoded Python dicts into versioned markdown files, with a build-time pipeline that generates both the MCP tool and the Claude Code skill from the same source. This is a POC — nothing in the existing `explain` tool is changed.

---

## Source Structure

```
skills/explain/
├── SKILL.md.template       ← human-authored; static skill instructions with {{TOPIC_REGISTRY}} placeholder
└── resources/
    ├── code-node-patterns.md          ← flat topic
    └── xapp/
        ├── index.md                   ← xapp primer (new topic: "xapp")
        ├── delivery.md                ← xapp-delivery
        └── event-handling.md         ← xapp-event-handling
```

The `resources/` directory is the authoritative source of truth. File path is organisational only — topic identity is defined by frontmatter, not path.

---

## Resource File Format

Each `.md` file carries YAML frontmatter:

```yaml
---
topic: xapp-delivery
description: session init, postMessage bridge, SDK.submit, dual xApp moments
group: xapp
---

## xapp-delivery — xApp Patterns
...markdown body...
```

**Fields:**

| Field | Required | Purpose |
|-------|----------|---------|
| `topic` | yes | Lookup key used by `explain_dev("xapp-delivery")`. Canonical identity. |
| `description` | yes | One-line entry in the topic index returned by `explain_dev()` with no args. |
| `group` | no | Logical cluster label for index display. Flat topics omit it. Topics sharing a `group` are listed together under a heading. |

The body (everything after the closing `---`) is the full topic content. No other structure required.

## Build Script

**Location:** `scripts/build_explain_topics.py`

**Invocation:** `uv run scripts/build_explain_topics.py`

**Inputs:** `skills/explain/resources/**/*.md` + `skills/explain/SKILL.md.template`

**Outputs:** two generated files, both committed to the repo.

### Pass

1. Recursively find all `*.md` under `resources/`
2. Parse frontmatter from each file (stdlib `re` — no external YAML dependency)
3. Validate: `topic` and `description` are required; abort with clear error if missing
4. Sort: flat topics first (no `group`), then grouped topics sorted by `group` then `topic`
5. Generate `_explain_topics_generated.py`
6. Render `SKILL.md.template` → `SKILL.md`

### Generated Python: `cognigy-mcp/cognigy_mcp/tools/_explain_topics_generated.py`

```python
# AUTO-GENERATED by scripts/build_explain_topics.py — do not edit directly
# Source: skills/explain/resources/

TOPICS: list[str] = ["code-node-patterns", "xapp", "xapp-delivery", "xapp-event-handling"]

_TOPIC_INDEX = """
Topics and what they cover:

  code-node-patterns   api.* functions, as const bug, httpRequest .result, no fetch/import

xapp:
  xapp                 overview of xApp architecture and variant selection
  xapp-delivery        session init, postMessage bridge, SDK.submit, dual xApp moments
  xapp-event-handling  non-blocking xApp pattern: flow structure, ephemeral data capture, SDK.submit vs webhook inject variants
"""

_CONTENT: dict[str, str] = {
    "code-node-patterns": "...",
    "xapp": "...",
    "xapp-delivery": "...",
    "xapp-event-handling": "...",
}
```

### Generated Skill: `skills/explain/SKILL.md`

The template `SKILL.md.template` contains the static skill instructions (when to use, how Claude should behave, how to read a resource file) with a `{{TOPIC_REGISTRY}}` placeholder. The build script substitutes the generated topic list — same content as `_TOPIC_INDEX` — and writes `SKILL.md`.

`SKILL.md` is committed alongside `SKILL.md.template`. The template is the human-editable source; `SKILL.md` is what Claude Code loads at runtime.

## MCP Changes

### New tool: `explain_dev`

Added to `cognigy-mcp/cognigy_mcp/tools/explain.py` alongside the existing `explain` tool.

```python
from cognigy_mcp.tools._explain_topics_generated import TOPICS as DEV_TOPICS, _TOPIC_INDEX as DEV_INDEX, _CONTENT as DEV_CONTENT
```

Behaviour is identical to `explain`:
- `explain_dev()` → returns `DEV_INDEX` (topic orientation)
- `explain_dev("xapp-delivery")` → returns `DEV_CONTENT["xapp-delivery"]`
- Unknown topic → clear error listing available topics from `DEV_TOPICS`

The existing `explain` tool, its `_CONTENT` dict, and `_TOPIC_INDEX` string are not touched.

### Tool registration

`explain_dev` is registered in the same `TOOLS` list and `make_handlers` dict as `explain`. No changes to server startup or routing.

---

## SKILL.md.template Design

The template covers:

- **When to use** — use `explain_dev` before brute-forcing or web-searching for implementation guidance on a covered topic
- **How to invoke** — `explain_dev()` for orientation, `explain_dev("topic")` for full reference
- **`{{TOPIC_REGISTRY}}`** — replaced at build time with the generated topic list
- **Gap behaviour** — if the topic isn't in `explain_dev`, fall back to `explain`

| File | Topic | Status |

| `resources/code-node-patterns.md` | `code-node-patterns` | Content from existing `_CONTENT` dict |

## What This Does Not Do

- Migrate any other existing topics — those remain in the hardcoded `_CONTENT` dict under `explain`
- Replace the `explain` tool — it runs unchanged in production
- Add CI validation that the generated files are up to date — deferred to migration issue

## Build / Publish Workflow

### Local (before committing topic changes)

```bash
uv run scripts/build_explain_topics.py
git add skills/explain/SKILL.md cognigy-mcp/cognigy_mcp/tools/_explain_topics_generated.py
git commit -m "chore: regenerate explain topics"
```

The generated files are committed to the repo so diffs are reviewable in PRs.

### GitHub Actions

The build script runs automatically in the existing `publish.yml` workflow, before the `uv build` step. This ensures the published wheel always contains up-to-date generated content regardless of whether local generation was run before commit.

**Change to `.github/workflows/publish.yml`** — add a step between the `setup-uv` action and `Build package`:

```yaml
- name: Generate explain topics
  if: steps.check.outputs.skip == 'false'
  run: uv run scripts/build_explain_topics.py
```

The script runs from the repo root (where `skills/` and `cognigy-mcp/` both live). The existing `cd cognigy-mcp` in the `Build package` step is unaffected — the generated Python file is already in place at `cognigy-mcp/cognigy_mcp/tools/_explain_topics_generated.py` when `uv build` runs.

`uv` is already available at this point via the `setup-uv` step. No additional setup required.

### Staleness check (deferred)

CI does not currently validate that committed generated files match what the script would produce. Adding a `git diff --exit-code` check after the script is a follow-on for the migration issue — it would fail the build if someone edits a `.md` file without regenerating.
