# Design: Explain Skill Structure + MCP Build Pipeline (Issue #44)

## Goal

Prove the end-to-end architecture for externalising `explain` topic content from hardcoded Python dicts into versioned markdown files, with a build-time pipeline that generates both the MCP tool and the Claude Code skill from the same source. This is a POC ‚Äî nothing in the existing `explain` tool is changed.

---

## Source Structure

```
skills/explain/
‚îú‚îÄ‚îÄ SKILL.md.template       ‚Üê human-authored; static skill instructions with {{TOPIC_REGISTRY}} placeholder
‚îî‚îÄ‚îÄ resources/
    ‚îú‚îÄ‚îÄ code-node-patterns.md          ‚Üê flat topic
    ‚îî‚îÄ‚îÄ xapp/
        ‚îú‚îÄ‚îÄ index.md                   ‚Üê xapp primer (new topic: "xapp")
        ‚îú‚îÄ‚îÄ delivery.md                ‚Üê xapp-delivery
        ‚îî‚îÄ‚îÄ event-handling.md         ‚Üê xapp-event-handling
```

The `resources/` directory is the authoritative source of truth. File path is organisational only ‚Äî topic identity is defined by frontmatter, not path.

---

## Resource File Format

Each `.md` file carries YAML frontmatter:

```yaml
---
topic: xapp-delivery
description: session init, postMessage bridge, SDK.submit, dual xApp moments
group: xapp
---

## xapp-delivery ‚Äî xApp Patterns
...markdown body...
```

**Fields:**

| Field | Required | Purpose |
|-------|----------|---------|
| `topic` | yes | Lookup key used by `explain_dev("xapp-delivery")`. Canonical identity. |
| `description` | yes | One-line entry in the topic index returned by `explain_dev()` with no args. |
| `group` | no | Logical cluster label for index display. Flat topics omit it. Topics sharing a `group` are listed together under a heading. |

The body (everything after the closing `---`) is the full topic content. No other structure required.

## Build Script

**Location:** `scripts/build_explain_topics.py`

**Invocation:** `uv run scripts/build_explain_topics.py`

**Inputs:** `skills/explain/resources/**/*.md` + `skills/explain/SKILL.md.template`

**Outputs:** two generated files, both committed to the repo.

### Pass

1. Recursively find all `*.md` under `resources/`
2. Parse frontmatter from each file (stdlib `re` ‚Äî no external YAML dependency)
3. Validate: `topic` and `description` are required; abort with clear error if missing
4. Sort: flat topics first (no `group`), then grouped topics sorted by `group` then `topic`
5. Generate `_explain_topics_generated.py`
6. Render `SKILL.md.template` ‚Üí `SKILL.md`

### Generated Python: `cognigy-mcp/cognigy_mcp/tools/_explain_topics_generated.py`

```python
# AUTO-GENERATED by scripts/build_explain_topics.py ‚Äî do not edit directly
# Source: skills/explain/resources/

TOPICS: list[str] = ["code-node-patterns", "xapp", "xapp-delivery", "xapp-event-handling"]

_TOPIC_INDEX = """
Topics and what they cover:

  code-node-patterns   api.* functions, as const bug, httpRequest .result, no fetch/import

xapp:
  xapp                 overview of xApp architecture and variant selection
  xapp-delivery        session init, postMessage bridge, SDK.submit, dual xApp moments
  xapp-event-handling  non-blocking xApp pattern: flow structure, ephemeral data capture, SDK.submit vs webhook inject variants
"""

_CONTENT: dict[str, str] = {
    "code-node-patterns": "...",
    "xapp": "...",
    "xapp-delivery": "...",
    "xapp-event-handling": "...",
}
```

### Generated Skill: `skills/explain/SKILL.md`

The template `SKILL.md.template` contains the static skill instructions (when to use, how Claude should behave, how to read a resource file) with a `{{TOPIC_REGISTRY}}` placeholder. The build script substitutes the generated topic list ‚Äî same content as `_TOPIC_INDEX` ‚Äî and writes `SKILL.md`.

`SKILL.md` is committed alongside `SKILL.md.template`. The template is the human-editable source; `SKILL.md` is what Claude Code loads at runtime.

## MCP Changes

### New tool: `explain_dev`

Added to `cognigy-mcp/cognigy_mcp/tools/explain.py` alongside the existing `explain` tool.

```python
from cognigy_mcp.tools._explain_topics_generated import TOPICS as DEV_TOPICS, _TOPIC_INDEX as DEV_INDEX, _CONTENT as DEV_CONTENT
```

Behaviour is identical to `explain`:
- `explain_dev()` ‚Üí returns `DEV_INDEX` (topic orientation)
- `explain_dev("xapp-delivery")` ‚Üí returns `DEV_CONTENT["xapp-delivery"]`
- Unknown topic ‚Üí clear error listing available topics from `DEV_TOPICS`

The existing `explain` tool, its `_CONTENT` dict, and `_TOPIC_INDEX` string are not touched.

### Tool registration

`explain_dev` is registered in the same `TOOLS` list and `make_handlers` dict as `explain`. No changes to server startup or routing.

---

## SKILL.md.template Design

The template covers:

- **When to use** ‚Äî use `explain_dev` before brute-forcing or web-searching for implementation guidance on a covered topic
- **How to invoke** ‚Äî `explain_dev()` for orientation, `explain_dev("topic")` for full reference
- **`{{TOPIC_REGISTRY}}`** ‚Äî replaced at build time with the generated topic list
- **Gap behaviour** ‚Äî if the topic isn't in `explain_dev`, fall back to `explain`

## POC Scope

Four resource files across two directory levels:

| File | Topic | Status |
|------|-------|--------|
| `resources/code-node-patterns.md` | `code-node-patterns` | Content from existing `_CONTENT` dict |
| `resources/xapp/index.md` | `xapp` | New primer ‚Äî written for this POC |
| `resources/xapp/delivery.md` | `xapp-delivery` | Content from existing `_CONTENT` dict |
| `resources/xapp/event-handling.md` | `xapp-event-handling` | Content from existing `_CONTENT` dict |

This exercises both flat and nested organisation in a single POC.

---

## What This Does Not Do

- Migrate any other existing topics ‚Äî those remain in the hardcoded `_CONTENT` dict under `explain`
- Replace the `explain` tool ‚Äî it runs unchanged in production
- Add CI validation that the generated files are up to date ‚Äî deferred to migration issue

## Build / Publish Workflow

Before cutting a PyPI release:

```bash
uv run scripts/build_explain_topics.py
git add skills/explain/SKILL.md cognigy-mcp/cognigy_mcp/tools/_explain_topics_generated.py
git commit -m "chore: regenerate explain topics"
uv build
```

This is documented in the repo but not yet enforced by CI (deferred to migration issue).
